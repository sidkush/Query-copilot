### Investigation Phase

We launched **6 parallel investigation agents**, each exploring a specific bug:

| Agent | Focus | Key Finding |
|-------|-------|-------------|
| 1 | Save not reflecting | `update_tile()` returns full dashboard, but frontend didn't always use it; added re-fetch safeguard |
| 2 | Tile sizing | Default `w:6 h:4` in 12-col grid with `rowHeight:80` = only 2 tiles/row, each 320px tall |

| 6 | Fullscreen mode | Doesn't exist; needs new feature |

### Fixes Implemented

#### Fix 1: Save Not Reflecting Instantly
**Root cause:** After `api.updateTile()`, the frontend tried to use the API response directly but had fragile fallback logic.

**Solution:** Changed both `handleTileSave` and `handleQuickTileUpdate` to always re-fetch the full dashboard after saving:

```javascript
// Before (fragile):
const savedDashboard = await api.updateTile(dash.id, updatedTile.id, updatedTile);
if (savedDashboard?.tabs) { setActiveDashboard(savedDashboard); }
else { /* manual state splice */ }

// After (robust):
await api.updateTile(dash.id, updatedTile.id, updatedTile);
const fresh = await api.getDashboard(dash.id);
if (fresh) setActiveDashboard(fresh);
```

**Files:** `DashboardBuilder.jsx` (lines 696-727, 778-804)

#### Fix 4: Pie Cross-Filter Blanks Other Charts
**Root cause:** The cross-filter logic blindly filtered ALL tiles by the clicked field. If a tile's data didn't contain that field (e.g., pie chart clicks "payment_method" but a line chart only has "date" and "revenue"), `row[crossFilter.field]` was undefined → zero rows matched → chart went blank.

**Solution:** Check if the field exists in the data before filtering:

```javascript
// Before:
return sortedData.filter(row => String(row[crossFilter.field]) === String(crossFilter.value));

// After:
const hasField = sortedData.length > 0 && crossFilter.field in sortedData[0];
if (!hasField) return sortedData; // Skip filter — field doesn't exist in this tile
return sortedData.filter(row => String(row[crossFilter.field]) === String(crossFilter.value));
```

**File:** `ResultsChart.jsx` (line 362-369)

#### Fix 5: Filtered Charts Lose Legend Colors
**Root cause:** Pie/donut charts used the filtered data's row index `i` for color resolution. After filtering (e.g., 8 slices → 2), indices 0-1 map to different palette colors than the original 0-7.

**Solution:** Built a `pieColorMap` from the unfiltered data that maps each label to its original index:

```javascript
const pieColorMap = useMemo(() => {
  const map = {};
  sortedData.forEach((row, i) => {
    const key = String(row[labelCol]);
    if (!(key in map)) map[key] = i;
  });
  return map;
}, [sortedData, labelCol]);

// In Cell rendering:
const origIdx = pieColorMap[String(entry[labelCol])] ?? i;
<Cell fill={resolveColor(currentMeasure, null, origIdx, fmt.colors, dashboardPalette)} />
```

**File:** `ResultsChart.jsx` (pie + donut cases)

#### Fix 6: Filter Tab
**Finding:** The `GlobalFilterBar` component was fully functional. It loads schema columns via `api.getTables(connId)`, lets users add field filters with operators (=, !=, >, <, LIKE, IN), and has an Apply button that triggers `handleGlobalFiltersChange`. The "not working" perception was likely due to needing an active database connection. No code changes were needed.

#### Fix 7: Export Not Working
**Root cause:** `document.getElementById('dashboard-content')` returned `null` because no element had that ID.

**Solution:** Added `id="dashboard-content"` to the sections content div in `DashboardBuilder.jsx`. Also improved the ExportModal with better error handling, PDF sizing using proper mm conversion, and temporary style overrides to capture full scrollable content.

**Files:** `DashboardBuilder.jsx` (line 1304), `ExportModal.jsx` (full rewrite of capture logic)

#### Fix 8: CommandBar Not Responding
**Root cause:** The frontend parsed `res?.tiles` but the backend returns a nested structure `res.tabs[].sections[].tiles[]`. The extraction always yielded an empty array, so the function returned early silently.

**Solution:** Extract tiles from the nested structure:

```javascript
// Before:
const tiles = res?.tiles || [];

// After:
const tiles = res?.tabs?.flatMap(tab =>
  (tab.sections || []).flatMap(sec => sec.tiles || [])
) || res?.tiles || [];
```

Also passed `activeConnId` to the API call (was passing `null` before).

**File:** `DashboardBuilder.jsx` (handleAICommand)

3. **FreeformCanvas.jsx** → TileWrapper: Added `themeConfig={themeConfig}`

- No visual feedback (user didn't know anything was happening)

## 4. Round 3 — CommandBar AI Search Engine

### Request

> The search bar in dashboard does not return anything. Fix it. It is an AI engine and should respond as an AI companion and return result in a chart created by adding a new tile in the dashboard.

### Deep Investigation

A thorough agent traced the **complete end-to-end flow** across 5 files:

```
User types "Show me revenue by month" → Enter
  ↓
CommandBar.handleSubmit → onAICommand(query)
  ↓
DashboardBuilder.handleAICommand(command)
  ↓
api.generateDashboard(command, connId) → POST /api/queries/generate-dashboard
  ↓
query_routes.py → get_connection(conn_id) → engine.generate_dashboard(request)
  ↓
query_engine.py:
  1. Retrieves schema from ChromaDB (RAG)
  2. Builds prompt with schema context
  3. Calls Claude API (Sonnet/Haiku) to generate SQL for multiple tiles
  4. Parses JSON response with repair logic
  5. EXECUTES each tile's SQL against the database
  6. Returns tiles with populated rows + columns
  ↓
Frontend receives tiles with data → addTileToSection for each → dashboard updates
```

**The backend was working correctly.** The issues were all on the frontend:

### Root Causes Found

1. **No section = instant silent bail:** `if (!targetSectionId) return;` — if the active tab had no sections, the function returned without any feedback
2. **No connection = silent 400 error:** The API call failed but the error was swallowed by `catch { console.error(...) }`
3. **No loading state:** User had zero feedback that anything was happening (AI generation takes 5-15 seconds)
4. **No error feedback:** All failures silently logged to console

### Complete Rewrite of handleAICommand

**Before:** 30 lines, no loading state, no error handling, silent failures everywhere.

**After:** 60 lines with:

1. **Loading state:** `aiCommandLoading` state drives a spinning indicator in CommandBar
2. **Auto-section creation:** If no section exists in the active tab, automatically creates an "AI Generated" section before adding tiles
3. **Connection validation:** Detects "No active database" errors and shows specific message
4. **Error display:** Red inline banner in CommandBar with auto-dismiss after 8 seconds
5. **Dashboard re-fetch:** After adding tiles, re-fetches full dashboard for guaranteed consistency

### CommandBar UI Overhaul

Rewrote `CommandBar.jsx` to accept new props:

- `aiLoading` — shows spinning indicator, disables input, shows "Generating charts..." placeholder
- `aiError` — shows red error banner below the search bar with dismiss button
- `onClearError` — callback to dismiss error (also auto-dismisses after 8s)
- Input disabled during loading to prevent double-submission

### Build Result
**0 errors, 1.10s build time**

## 5. Round 4 — Tile Height Normalization

### Request

> Revenue charts in KPI field are too long in height. Reduce their height to match the above ones as in the picture. The container sizes of newly added tiles are too large in height.

### Analysis

The height budget per tile with current grid settings:
- Grid: `rowHeight: 60`, default tile `h: 3` → **180px total cell height**
- KPI cards: ~100px content, compact, fits perfectly
- Chart tiles: Header (~40px) + Chart (240px) + Summary bar (~30px) + Padding (~36px) = **~346px** — far exceeding 180px

The chart content was overflowing the grid cell due to `overflow-visible` on TileWrapper.

### Fixes Applied

| Component | Property | Before | After | Savings |
|-----------|----------|--------|-------|---------|
| `ResultsChart` | Chart height (embedded) | 240px | **150px** | -90px |
| `ResultsChart` | Chart canvas padding | `px-2 py-4` | `px-1 py-1` | -24px |
| `ResultsChart` | Data summary bar | Always shown | **Hidden in embedded** | -30px |
| `TileWrapper` | Chart body min-height | 160px | **80px** | -80px |
| `TileWrapper` | Chart body padding | `12px 18px 18px` | `4px 12px 8px` | -18px |
| `TileWrapper` | Header padding | `px-[18px] pt-[14px]` | `px-[14px] pt-[10px]` | -4px |

**Total height reduction per chart tile: ~150px.** Chart tiles now render at ~196px, fitting within the h:3 grid cell (180px + 12px margin tolerance) and visually matching KPI card heights.

## 7. Architecture Decisions

### Decision 1: Re-fetch After Every Mutation
**Pattern:** After any tile save, quick update, or refresh, we now call `api.getDashboard(id)` to re-fetch the full dashboard instead of relying on the API response or manual state splicing.

**Why:** The previous approach had fragile fallback logic that checked `response?.tabs` and fell through to manual state updates. Different endpoints returned different response shapes. The re-fetch pattern is slower by one API call but guarantees state consistency.

### Decision 2: Fullscreen as Component State, Not Route
**Pattern:** `fullscreenMode` is a boolean state in DashboardBuilder, not a separate route.

**Why:** Preserves all dashboard state (active tab, filters, cross-filters, scroll position) without URL changes or re-mounting. Escape key exits cleanly. The sidebar is conditionally hidden via `{!fullscreenMode && <aside>...</aside>}`.

### Decision 3: pieColorMap for Stable Cross-Filter Colors
**Pattern:** Build a `Map<label, originalIndex>` from unfiltered data, use original index for color resolution after filtering.

**Why:** The simpler approach of using `chartData.indexOf()` wouldn't work because filtered data changes the iteration order. The map approach preserves the exact palette assignment from the initial render.

### Decision 4: Auto-Section Creation in AI Command
**Pattern:** If the active tab has no sections, `handleAICommand` creates one before adding tiles.

**Why:** The previous code silently returned if no section existed. Since AI-generated tiles need a container, auto-creating one with a descriptive name ("AI Generated") is the right UX.
