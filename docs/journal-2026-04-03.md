## 2. Phase 1 — Git Hygiene & GitHub Push

### Problem
The repository had never been pushed to GitHub. Worse, `backend/.env` and `backend/gcp-credentials.json` containing API keys, JWT secrets, and GCP credentials were tracked in git history across 18 commits.

### Bottleneck: Secrets Baked Into History
Simply adding a `.gitignore` and running `git rm --cached` removes files from the current tree but leaves them in every historical commit. GitHub's push protection caught this and blocked the push.

   ```
   python -m git_filter_repo --path backend/.env --path backend/gcp-credentials.json \
     --path backend/.data/admin_credentials.json --invert-paths --force
   ```

4. Re-added remote (filter-repo strips remotes as a safety measure)
5. Force-pushed to GitHub after user approval

### Thinking Process
Considered three options: (a) nuke history and start fresh (loses context), (b) BFG Repo Cleaner (requires Java), (c) git-filter-repo (Python-native, already in our stack). Chose (c) because it's the cleanest tool for the job and the user explicitly wanted to keep login state and tokens intact locally.

## 4. Phase 3 — Tile Sizing: The Spread Order Bug

### Problem
After the initial fix, the user reported tiles still couldn't shrink to match KPI tile sizes. "Total Revenue" and "Total Customers" (KPI, h=1) were small, but chart tiles (bar, line, donut) were stuck at h=3 minimum.

### Root Cause Investigation (ultraflow debugging, Phase 1-4)

**Observation:** Queried the saved dashboard JSON:
```
tile='Daily Revenue'  type=area  w=6 h=4 minW=3 minH=3
tile='Total Revenue'  type=kpi   w=3 h=1 minW=2 minH=1
```

Every chart tile had `minH: 3` **baked into the saved layout data**.

**Hypothesis 1 (confirmed):** The spread order `{ minW: 2, minH: 2, ...item }` puts defaults BEFORE saved data. Since saved items have `minH: 3`, the spread produces `{ minW: 2, minH: 2, minH: 3 }` → `minH: 3` wins.

**Hypothesis 2 (confirmed):** TileWrapper's chart body had `min-h-[80px]` Tailwind class creating an additional CSS floor.

### Fix
1. **Reversed spread order:** `{ ...item, minW: 2, minH: 1 }` — our overrides come AFTER saved data, enforcing `minH: 1` (60px, matching KPI tiles)
2. **Removed CSS floor:** Changed `min-h-[80px]` to `min-h-0` on chart body div
3. **Lowered freeform minimums:** 100x80 -> 80x60

### Thinking Process
The key insight was checking the **actual persisted data**, not just the code. The spread `{ defaults, ...saved }` vs `{ ...saved, overrides }` distinction is subtle — the fix was literally swapping the order of two arguments. This is why reading data at rest is essential for debugging layout issues.

## 5. Phase 4 — Measure Checkboxes: The Three-Layer Disconnect

### Problem
In the TileEditor, all measure checkboxes appeared unchecked, yet the chart showed all measures. Checking/unchecking and saving didn't update the chart.

### Root Cause Investigation

**Three independent bugs creating one symptom:**

| Layer | Bug | Effect |
|-------|-----|--------|
| TileEditor (init) | `useState(tile.activeMeasures \|\| [])` | Empty array when no measures explicitly saved → all unchecked |
| ResultsChart (init) | `useState(defaultMeasures \|\| numericCols)` | Falls back to ALL columns when `defaultMeasures` is `[]` → shows everything |
| ResultsChart (update) | `useState(prop)` with no sync | Prop changes after save ignored — stale internal state |

Plus: The `key` prop on `<ResultsChart>` didn't include `activeMeasures`, so React never remounted the component when measures changed.

### Fix
1. **TileEditor:** Init `activeMeasures` to `tile.activeMeasures?.length ? tile.activeMeasures : [...tile.columns]` — checkboxes now reflect what's actually visible
2. **ResultsChart:** Added `useEffect` to sync `activeMeasures` from `defaultMeasures` prop changes
3. **ResultsChart:** Changed init to `defaultMeasures?.length ? defaultMeasures : numericCols` — empty array now correctly treated as "no explicit selection"
4. **TileWrapper:** Added `activeMeasures` to the `key` prop, forcing remount on measure changes

### Thinking Process
This was a classic case of React's `useState(prop)` anti-pattern. The state initializes from the prop but never updates when the prop changes. The fix required understanding the full data flow: TileEditor state → save API → re-fetch dashboard → new props to TileWrapper → ResultsChart. Each layer had its own assumption about what "empty" meant.

## 8. Phase 7 — Filter Pipeline: Killing the Reactive Chain

### Problem
After all previous fixes, the GlobalFilterBar still didn't filter tiles. Selecting "This Week" or adding field filters and clicking "Apply" produced no visible change.

### Root Cause Investigation (the deepest debugging session)

This was the hardest bug of the day. The reactive chain was:

```
User clicks Apply
  → handleGlobalFiltersChange(newFilters)
    → applyGlobalFilters(newFilters)     [bumps dashboardFilterVersion in Zustand]
    → autoSave(...)                       [persists filters]
  → useEffect detects version change
    → 300ms debounced setTimeout
      → handleTileRefresh for each tile
        → api.refreshTile (backend applies filters)
        → api.getDashboard (re-fetch full dashboard)
        → setActiveDashboard(fresh)
```

**Initial investigation:** Traced every step of the chain. Each piece looked correct in isolation:
- GlobalFilterBar sends `{ dateColumn, range, dateStart, dateEnd, fields }`
- Zustand stores it correctly with version bump
- Backend correctly wraps SQL with date filters and field conditions
- `resolveColor`, `mergeFormatting` all correct

**The fragility:** The reactive useEffect chain had multiple failure modes:
1. **Cleanup race:** Any re-render between the version bump and the 300ms timeout could trigger useEffect cleanup, clearing the debounced timer. The re-run would see the version refs already updated and exit early — **the refresh silently never happens.**
2. **Concurrent race:** `Promise.allSettled` fired all tile refreshes concurrently. Each called `api.getDashboard` then `setActiveDashboard`. Early `getDashboard` responses were fetched before later tiles finished saving, creating stale overwrites.
3. **Ref staleness:** `globalFiltersRef.current` assignment during render could race with the useEffect reading it during the timeout callback.

**The breakthrough:** Rather than trying to fix the timing of the reactive chain, we realized the indirection was the problem itself. Reactive version-counter patterns work well for cross-component communication but are overkill when the trigger and effect are in the same component. A direct function call is simpler and more reliable.

### Fix
1. **Replaced reactive chain with direct refresh:** `handleGlobalFiltersChange` now directly calls `refreshAllTiles(newFilters)` — no version counters, no debounce, no useEffect intermediary
2. **Sequential tile refresh:** Changed from `Promise.allSettled` (concurrent) to sequential `for...of` loop. Each tile completes its refresh cycle (refreshTile → getDashboard → setActiveDashboard) before the next starts. Eliminates the stale-overwrite race condition.
3. **Simplified tileEditVersion effect:** The useEffect now only handles tile-save-triggered refreshes (single responsibility)
4. **Removed dead code:** `dashboardFilterVersion` store subscription, `filterVersionRef`, `refreshDebounceRef`

### Thinking Process
This was a lesson in **over-engineering**. The reactive Zustand architecture was the right choice for the store design (future-proof, clean separation), but using version counters + useEffect + debounce for same-component communication added three layers of indirection where a function call sufficed. The fix was to keep the Zustand store for persistence/sharing but use direct function calls for the refresh trigger.

## 10. Architecture Decisions & Trade-offs

### Decision: Reactive Zustand vs Surgical Fixes
**Chose reactive.** Surgical fixes would have been faster for the immediate bugs, but the dashboard is the monetization core. Future features (cross-dashboard filters, real-time collaboration, undo/redo) need a centralized state layer. The Zustand slices are that foundation.

### Decision: Direct Refresh vs Reactive useEffect for Filters
**Chose direct.** The reactive useEffect was elegant in theory but fragile in practice. React's render cycle creates timing windows where cleanup functions can kill pending operations. Direct function calls are harder to break.

### Decision: Sequential vs Concurrent Tile Refresh
**Chose sequential.** Concurrent is faster but creates race conditions when multiple `setActiveDashboard` calls overwrite each other. Sequential is slower but guarantees each tile's filtered data is preserved. For dashboards with many tiles, a future optimization could batch the final `getDashboard` call after all refreshes complete.

### Decision: Layout Apply Button vs Auto-Save
**Chose explicit apply.** Auto-save on every drag creates accidental persistence. The user should experiment freely and commit only when satisfied. This matches the mental model of "edit mode" vs "view mode."

## 12. Lessons Learned

### 1. Check the data, not just the code
The tile sizing bug was invisible in the code. The spread `{ minW: 2, minH: 2, ...item }` looked correct until we queried the actual saved JSON and found `minH: 3` in every chart tile. **Always inspect persisted state when debugging layout issues.**

### 2. `useState(prop)` is a React footgun
The measure checkbox bug was caused by `useState(defaultMeasures || numericCols)` — a pattern that initializes from a prop but never syncs. This is one of React's most common anti-patterns. Either use the prop directly (derived state) or add a useEffect to sync.

### 3. Indirection has a cost
The reactive version-counter pattern (store bump → useEffect → debounce → refresh) added three layers of indirection. Each layer introduced a timing window where React's render cycle could interrupt the chain. A direct function call eliminated all three failure modes at once.

### 4. Sequential beats concurrent when state is shared
`Promise.allSettled` for 10 tile refreshes is faster, but each refresh calls `setActiveDashboard` which overwrites the previous. The last response wins, and it might contain stale data for tiles that refreshed earlier. Sequential refresh is slower but correct.

### 5. Spread order matters
`{ defaults, ...saved }` means saved wins. `{ ...saved, overrides }` means overrides win. A one-character difference in where `...item` goes completely changes the behavior. This is easy to get wrong and hard to spot in code review.

### 6. Hidden UI is broken UI
The per-measure color picker existed and worked perfectly — but it was hidden behind Colors tab → "custom" mode toggle. Three clicks deep. Moving it inline next to each measure checkbox made it instantly discoverable. **If users can't find a feature, it doesn't exist.**

### 7. git-filter-repo is the right tool for secret scrubbing
`git rm --cached` only removes from the current tree. Secrets persist in every historical commit. `git-filter-repo` rewrites the entire history cleanly. Always scrub before the first push, not after.

## 14. Phase 9 — Cumulative Revenue Growth: Subquery SQL Injection Bug

### Problem
One tile — "Cumulative Revenue Growth" — consistently failed to refresh with filters. Its SQL used a nested subquery with a window function:
```sql
SELECT DATE(o.created_at) AS day,
       SUM(oi.quantity * oi.unit_price) AS daily_revenue,
       SUM(SUM(oi.quantity * oi.unit_price)) OVER (ORDER BY DATE(o.created_at)) AS cumulative
FROM orders o JOIN order_items oi ON o.id = oi.order_id
GROUP BY DATE(o.created_at) ORDER BY day
```

### Where We Got Stuck
The original `_inject_date_filter` function in `dashboard_routes.py` found the column reference `o.created_at`, then scanned forward for the first `ORDER BY` or `GROUP BY` as the injection point. But it found `ORDER BY` inside `OVER(ORDER BY DATE(o.created_at))` FIRST — a window function clause, not the main query's ORDER BY.

**Result:** The injected SQL became:
```sql
SUM(SUM(...)) OVER ( WHERE o.created_at >= '2026-03-23' ORDER BY ...
```
A syntactically broken query — WHERE clause injected inside a window function.

### First Fix Attempt (Failed)
Tried a depth-tracking approach: count parentheses from the column position forward, only consider `ORDER BY`/`GROUP BY` at depth 0. But `DATE(o.created_at)` added +1 depth, and the closing `)` of `DATE()` restored depth to 0 at the wrong position. `GROUP BY` was found at depth 1 (inside the outer `OVER(...)`) instead of depth 0.

### Why It Failed
The depth counter couldn't distinguish between:
- `DATE(...)` — a function call paren (should be "transparent")
- `OVER(...)` — a window function paren (defines a scope)
- `(SELECT ...)` — a subquery paren (defines a scope)

All three use `(` and `)` — the character-counting approach had no semantic awareness.

### Brainstorm: Walk-Backwards Subquery Detection
Instead of counting forward from the column, we walk BACKWARDS from the column position to find the enclosing `(SELECT ...)` subquery:

1. Start at `col_pos - 1`, walk left
2. Track depth: `)` increments, `(` decrements
3. When we hit `(` at depth 0, check if the text after it starts with `SELECT`
4. If yes → this is a subquery boundary. Find its matching `)` going forward.
5. If no → this is a function call paren (`DATE(`, `SUM(`). Skip it and keep walking.

This elegantly solves the problem: `DATE(...)` parens are skipped because the text after `(` is `o.created_at`, not `SELECT`. `OVER(...)` parens are skipped because text after `(` is `ORDER BY`, not `SELECT`. Only actual subquery parens match.

### Final Fix
```python
def _inject_date_filter(sql, col, start, end):
    # Walk backwards from column to find enclosing (SELECT ...) scope
    col_pos = re.search(re.escape(col), sql, re.IGNORECASE).start()
    scope_start, scope_end = 0, len(sql)
    depth = 0
    for i in range(col_pos - 1, -1, -1):
        if sql[i] == ')': depth += 1
        elif sql[i] == '(':
            if depth == 0:
                rest = sql[i + 1:].lstrip()
                if rest.upper().startswith('SELECT'):
                    scope_start = i + 1
                    # Find matching closing paren
                    d = 1
                    for j in range(i + 1, len(sql)):
                        if sql[j] == '(': d += 1
                        elif sql[j] == ')':
                            d -= 1
                            if d == 0: scope_end = j; break
                    break
            else: depth -= 1
    # Inject within the identified scope
    scope_sql = sql[scope_start:scope_end]
    boundary = re.search(r'\b(GROUP\s+BY|ORDER\s+BY|HAVING|LIMIT)\b', scope_sql, re.IGNORECASE)
    # ... inject WHERE/AND before boundary or at scope end
```

### Verification
Tested via curl with the actual Cumulative Revenue Growth SQL → returned valid filtered results.

## 15. Phase 10 — Cross-Tab Filter Propagation

### Problem
Filters applied on Tab 0 (Revenue Overview) didn't affect tiles on Tab 1 (Campaign & Channel Analysis) or Tab 2 (Product & Customer Revenue). Switching tabs showed unfiltered data.

### Where We Got Stuck
The `refreshAllTiles` function only iterated tiles from the CURRENT tab. When the user applied filters on Tab 0 and then switched to Tab 1, the Tab 1 tiles had never been refreshed with filters.

### Fix: Tab-Switch Auto-Refresh
Added a `useEffect` in `DashboardBuilder.jsx` that watches `activeTabId`:
```jsx
const prevTabIdRef = useRef(activeTabId);
useEffect(() => {
  if (activeTabId === prevTabIdRef.current) return;
  prevTabIdRef.current = activeTabId;
  const filters = globalFiltersRef.current;
  const hasDateFilter = filters?.dateColumn && filters?.range && filters.range !== 'all_time';
  const hasFieldFilters = filters?.fields?.length > 0;
  if (hasDateFilter || hasFieldFilters) {
    refreshAllTiles(filters);
  }
}, [activeTabId, refreshAllTiles]);
```

When the user switches tabs with active filters, all tiles in the new tab get refreshed with the current filter state. No manual re-apply needed.

## 16. Phase 11 — Campaign Tiles: The Date Semantics Problem

### Problem
After cross-tab propagation was fixed, Campaign & Channel Analysis tiles returned 0 rows when filtered to "Last Week." The error message showed "No data for selected filters" — technically working, but semantically wrong.

### Where We Got Stuck (Longest Debugging Detour)
Campaign tiles don't have `created_at`. The fallback logic tried common date columns and found `start_date`. It injected:
```sql
WHERE c.start_date >= '2026-03-23' AND c.start_date <= '2026-03-29'
```

But NO campaigns STARTED during that week. Campaigns have `start_date` = 2025-10-01 to 2026-03-10. The filter was technically correct but conceptually wrong.

### The Insight: Two Different Date Models
We realized the dashboard had two fundamentally different date models:

| Table | Date Model | Semantics |
|-------|-----------|-----------|
| `orders` | Point-in-time (`created_at`) | Event happened at this moment |
| `campaigns` | Duration (`start_date` → `end_date`) | Active during this period |

Filtering orders by "Last Week" means: orders created during that week.
Filtering campaigns by "Last Week" means: campaigns that were **active** during that week — NOT campaigns that started during that week.

### Brainstorm: Overlap Semantics
A campaign is "active during [start, end]" if:
```
campaign.start_date <= range_end AND (campaign.end_date >= range_start OR campaign.end_date IS NULL)
```
The `IS NULL` handles ongoing campaigns with no end date.

This is the classic interval overlap test from computational geometry.

### Fix
1. Created `_inject_date_filter_overlap(sql, start_col, end_col, start, end)` — same scope-aware injection as `_inject_date_filter`, but with overlap condition
2. Modified fallback detection: when `start_date` is the fallback column, also check for `end_date` in the SQL. If both exist, use overlap semantics:
```python
if fallback == 'start_date':
    end_col_for_overlap = _find_qualified_col(target_sql, 'end_date')
if end_col_for_overlap:
    target_sql = _inject_date_filter_overlap(
        target_sql, qualified_col, end_col_for_overlap, s_str, e_str)
else:
    target_sql = _inject_date_filter(target_sql, qualified_col, s_str, e_str)
```

### Verification
Curl test with "Last Week" filter on Campaign tiles → returned 5 rows (campaigns active during that week) instead of 0.

## 17. Phase 12 — Infinite Loading Spinner vs Empty State

### Problem
After the overlap fix, some tiles showed "Loading data..." forever instead of "No data for selected filters." The spinner never resolved.

### Root Cause
`TileWrapper.jsx` used this logic:
```jsx
chartRows.length === 0 && tile.sql → show spinner
```
It couldn't distinguish between:
- **Not loaded yet** → spinner is correct
- **Loaded but empty** → should show "No data" message

Both states had `chartRows.length === 0`.

### The Insight
When data has been fetched, the tile has `columns` populated (even if `rows` is empty). When data hasn't been fetched yet, `columns` is also empty.

### Fix
```jsx
// Fetched but empty (columns exist, rows don't)
tile?.sql && tile?.columns?.length > 0 ? (
  <div>No data for selected filters</div>
// Not yet fetched (no columns either)
) : tile?.sql ? (
  <div>Loading data...</div>
)
```

The presence of `columns` is the differentiator: the backend always returns column metadata even when zero rows match.

## 18. Architecture of the Final Filter Pipeline

After all 5 phases, the complete filter flow:

```
User selects date filter + clicks Apply (GlobalFilterBar)
  → handleGlobalFiltersChange(filters)
    → autoSave(dashboard)              [persist filters]
    → refreshAllTiles(filters)          [direct call, no reactive chain]
      → for each tile in current tab:
          → POST /api/dashboards/{id}/tiles/{tileId}/refresh
            → Backend: find date column in tile SQL
              → Primary: user-selected dateColumn
              → Fallback: scan _DATE_COL_CANDIDATES list
                → If start_date + end_date found: overlap semantics
                → Else: standard range filter
            → _inject_date_filter / _inject_date_filter_overlap
              → Walk backwards from column to find (SELECT ...) scope
              → Inject WHERE/AND before GROUP BY/ORDER BY/HAVING/LIMIT
            → Execute filtered SQL → return results
          → Frontend updates tile with new data
      → Failed tiles reported by name

User switches tab (with active filters)
  → useEffect detects activeTabId change
    → Checks if filters are active
    → refreshAllTiles(currentFilters) for new tab's tiles
```

## 19. Extended Lessons Learned

### 8. SQL injection is scope-sensitive
Injecting `WHERE` clauses into SQL isn't just "find GROUP BY and insert before it." Window functions (`OVER(ORDER BY ...)`), subqueries, and function calls all create nested scopes. Walking backwards to find the `(SELECT ...)` boundary is the only reliable approach for arbitrary SQL.

### 9. Different data models need different filter semantics
Point-in-time fields (`created_at`) need range filters: `col >= start AND col <= end`. Duration fields (`start_date`/`end_date` pairs) need overlap filters: `start <= range_end AND (end >= range_start OR end IS NULL)`. A one-size-fits-all date filter will silently return wrong results for half the data.

### 10. Empty state vs loading state need distinct signals
`rows.length === 0` is ambiguous — it could mean "hasn't loaded" or "loaded empty." Always provide a secondary signal (like `columns.length > 0`) to distinguish the two. This is a common frontend anti-pattern that creates phantom loading spinners.

### 11. Server reload on Windows requires kill-all
Windows Python processes don't always respond to `uvicorn --reload`. Stale `.pyc` caches and orphaned processes on the same port can silently serve old code. The reliable pattern: kill ALL Python processes → clear `__pycache__` → restart. We had to do this 4+ times during the session.

### 12. Test with the actual data, not assumed data
The campaign filter bug was invisible until we queried the actual database and found that no campaigns had `start_date` during "Last Week." The code was syntactically correct — the semantics were wrong. Always verify with real data ranges.

### 13. Direct calls beat reactive chains for same-component triggers
The version-counter → useEffect → debounce → refresh chain had three failure modes. Replacing it with a direct `refreshAllTiles()` call eliminated all of them. Reserve reactive patterns for cross-component communication; use direct calls when trigger and effect live in the same component.
